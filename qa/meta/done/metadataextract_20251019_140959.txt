# SAP SuccessFactors Internal Secondment Metadata Extraction Report

## Executive Summary
This metadata extraction analyzes the data model requirements for processing date-effective internal secondments within SAP SuccessFactors Employee Central, focusing on the automatic defaulting of employment details from secondment positions with HR/Payroll override capabilities.

## 1. Core Data Objects for Internal Secondment Processing

### 1.1 EmploymentInfo Object (Enhanced for Secondments)
```
Entity: EmploymentInfo
- Primary Key: employmentId
- Purpose: Track primary employment and secondment relationships
- Key Fields:
  - employmentType (Picklist: Primary, Secondment, Internal Transfer)
  - secondmentType (Picklist: Internal, External, Temporary)
  - secondmentStartDate (Date field for effective dating)
  - secondmentEndDate (Date field for secondment termination)
  - secondmentSourcePositionId (Reference to source position)
  - secondmentTargetPositionId (Reference to secondment position)
  - isSecondmentActive (Boolean flag)
  - secondmentStatus (Picklist: Pending, Active, Completed, Cancelled)
- Temporal Management: Full effective dating support for secondment periods
- Relationships: Person → EmploymentInfo → JobInfo (with secondment context)
```

### 1.2 JobInfo Object (Secondment-Specific Fields)
```
Entity: JobInfo
- Primary Key: jobInfoId
- Purpose: Store job details with secondment-specific overrides
- Key Fields:
  - jobCode (Inherited from position but overridable)
  - positionTitle (Defaulted from position, overridable by HR/Payroll)
  - businessUnit (Defaulted from position, overridable)
  - department (Defaulted from secondment position)
  - costCenter (May change during secondment)
  - jobLevel (Defaulted but overridable for secondment)
  - employmentType (Maintains secondment type)
  - isSecondmentOverride (Boolean: indicates manual override)
  - secondmentOverrideReason (Text field for audit trail)
- Temporal Management: Date-effective for secondment periods
```

### 1.3 Position MDF Object (Secondment Source/Target)
```
Entity: Position (MDF Object)
- Primary Key: positionId
- Purpose: Position information for secondment processing
- Key Fields:
  - positionTitle (Default value for secondment)
  - jobCode (Default job classification)
  - businessUnit (Default organizational assignment)
  - department (Default department assignment)
  - costCenter (Default cost center)
  - reportsToPositionId (Secondment reporting structure)
  - positionStatus (Active, Inactive, Frozen)
  - positionType (Picklist: Regular, Temporary, Secondment)
- Relationships: Position → JobInfo (for secondment defaults)
```

## 2. Internal Secondment MDF Objects

### 2.1 InternalSecondment (Custom MDF Object)
```
Entity: InternalSecondment (Custom MDF)
- Primary Key: secondmentId
- Purpose: Track internal secondment lifecycle
- Key Fields:
  - employeeId (Reference to Employee Central)
  - sourcePositionId (Original position reference)
  - targetPositionId (Secondment position reference)
  - secondmentStartDate (Effective start date)
  - secondmentEndDate (Planned/actual end date)
  - secondmentReason (Picklist: Development, Coverage, Special Project)
  - secondmentStatus (Picklist: Draft, Approved, Active, Completed, Cancelled)
  - hrOverrideFlag (Boolean: indicates HR override applied)
  - payrollOverrideFlag (Boolean: indicates Payroll override applied)
  - createdDate, createdBy, lastModified, lastModifiedBy
  - approvalWorkflowId (Reference to workflow instance)
- Temporal Management: Not date-effective (single secondment instance)
- Relationships: Employee → InternalSecondment → Position
```

### 2.2 SecondmentOverrideDetails (Custom MDF Object)
```
Entity: SecondmentOverrideDetails (Custom MDF)
- Primary Key: overrideId
- Purpose: Track specific overrides applied during secondment
- Key Fields:
  - secondmentId (Foreign key to InternalSecondment)
  - overrideField (Picklist: PositionTitle, Department, CostCenter, JobLevel, etc.)
  - defaultValue (Original value from target position)
  - overriddenValue (HR/Payroll modified value)
  - overriddenBy (User ID of HR/Payroll officer)
  - overrideDate (Date/time of override)
  - overrideReason (Text field for justification)
  - approvedBy (Approver for override)
  - approvedDate (Date of override approval)
- Relationships: InternalSecondment → SecondmentOverrideDetails
```

## 3. Foundation Objects for Secondment Configuration

### 3.1 Secondment Reason Foundation Object
```
Entity: SecondmentReason (FO)
- Primary Key: code
- Purpose: Standardized secondment reason codes
- Key Fields:
  - code (Unique identifier)
  - name (Display name)
  - description (Full description)
  - effectiveStartDate, effectiveEndDate (Temporal management)
  - status (Active/Inactive)
- Used in: InternalSecondment.secondmentReason field
```

### 3.2 Secondment Status Foundation Object
```
Entity: SecondmentStatus (FO)
- Primary Key: code
- Purpose: Standardized secondment status values
- Key Fields:
  - code (Status code)
  - name (Status display name)
  - description (Status explanation)
  - effectiveStartDate, effectiveEndDate
- Used in: InternalSecondment.secondmentStatus field
```

## 4. Picklist Metadata for Secondment Processing

### 4.1 Employment Type Picklist
```
Picklist: employmentType
- Values: 
  - PRIM (Primary Employment)
  - SECO (Secondment)
  - INTR (Internal Transfer)
  - TEMP (Temporary Assignment)
- Usage: EmploymentInfo.employmentType
- Default: PRIM for primary, SECO for secondments
```

### 4.2 Secondment Type Picklist
```
Picklist: secondmentType
- Values:
  - INT (Internal Secondment)
  - EXT (External Secondment)
  - TEMP (Temporary Assignment)
- Usage: EmploymentInfo.secondmentType
- Default: INT for internal secondments
```

### 4.3 Override Reason Picklist
```
Picklist: overrideReason
- Values:
  - SALARY_ADJUSTMENT (Salary changes during secondment)
  - DEPARTMENT_CHANGE (Department differs from position default)
  - COST_CENTER_CHANGE (Cost center override required)
  - JOB_LEVEL_ADJUSTMENT (Temporary level change)
  - SPECIAL_CONDITIONS (Special contractual terms)
- Usage: SecondmentOverrideDetails.overrideReason
```

## 5. OData API Endpoints for Secondment Data

### 5.1 Core Secondment Endpoints
```
OData Endpoints:
- /odata/v2/InternalSecondment - Custom MDF object for secondments
- /odata/v2/SecondmentOverrideDetails - Override tracking
- /odata/v2/Position - Target/source position information
- /odata/v2/JobInfo - Job details during secondment period
- /odata/v2/EmploymentInfo - Employment relationship tracking
- /odata/v2/SecondmentReason - Configuration picklist
- /odata/v2/SecondmentStatus - Status configuration
```

### 5.2 Query Examples for Secondment Processing
```
Query Examples:
- Active secondments: /odata/v2/InternalSecondment?$filter=secondmentStatus eq 'Active' and secondmentStartDate le $currentDate and (secondmentEndDate eq null or secondmentEndDate ge $currentDate)
- Secondments requiring override review: /odata/v2/InternalSecondment?$filter=hrOverrideFlag eq true
- Position defaults for secondment: /odata/v2/Position?$filter=positionId eq 'TARGET_POSITION_ID'
```

## 6. Business Rules and Workflows

### 6.1 Secondment Processing Business Rules
```
Business Rules:
- Default Value Rule: Automatically populate JobInfo fields from target Position MDF
- Date Validation Rule: Ensure secondment end date > start date
- Override Authorization: HR/Payroll user validation for override permissions
- Temporal Consistency: Validate no overlapping secondments
- Position Availability: Verify target position is active during secondment period
```

### 6.2 Workflow Integration Points
```
Workflow Triggers:
- Secondment creation (requires approval)
- Override application (HR/Payroll review)
- Secondment end date approaching (notification)
- Position change during secondment (revalidation)
```

## 7. Security and Permission Metadata

### 7.1 Role-Based Access for Secondment Processing
```
Permissions:
- HR Officer: Read/Write InternalSecondment, JobInfo overrides
- Payroll Officer: Read/Write override details, JobInfo modifications
- Manager: Read access to reports, approval capabilities
- Employee: Read access to own secondment details (self-service)
- Administrator: Full access to all secondment objects
```

### 7.2 Field-Level Security
```
Field Security:
- SecondmentOverrideDetails.overriddenValue: HR/Payroll only
- InternalSecondment.secondmentEndDate: HR/Payroll override capability
- JobInfo fields during secondment: Conditional override permissions
```

This metadata extraction provides the comprehensive data model foundation for implementing date-effective internal secondments with automatic defaulting and HR/Payroll override capabilities in SAP SuccessFactors Employee Central.
