# SAP SuccessFactors Metadata Extraction Report: Teacher Transfer Points System

## Executive Summary
This comprehensive metadata extraction focuses on the Teacher Transfer Points system within the Queensland Department of Education's HCMS program, specifically analyzing the BTP (Business Technology Platform) application for teacher transfers and the underlying data structures required for transfer point calculations. The analysis covers the custom solution architecture bypassing standard RCM limitations.

## Core Entity Analysis

### Primary Foundation Objects

```
Entity: FOCompany
- Purpose: Organizational structure for transfer rating locations
- Key Fields: 
  - companyCode (String, 32 char, Required)
  - companyName (String, 255 char, Required)
  - transferRatingLevel (Integer, 1-7, Required)
  - locationType (Picklist: URBAN, RURAL, REMOTE, METROPOLITAN)
  - effectiveStartDate (Date, Required)
  - effectiveEndDate (Date, Required)
- Relationships: Linked to JobInfo, EmploymentInfo for location-based calculations
- Validation: transferRatingLevel must be between 1-7 (inclusive)

Entity: FODepartment  
- Purpose: School/Regional office structures
- Key Fields: 
  - departmentCode (String, 32 char, Required)
  - departmentName (String, 255 char, Required)
  - transferRating (Integer, 1-7, Required)
  - schoolType (Picklist: PRIMARY, SECONDARY, COMBINED, SPECIAL)
  - regionCode (String, 16 char, Required)
  - isAcceleratedAccrual (Boolean, Default: false)
- Relationships: Connected to employee assignments and transfer history
- Validation: transferRating constrained to 1-7 scale
```

### Transfer Points Calculation Objects

```
  - transferId (String, 32 char, Required, Unique)
```


## Data Validation Rules

```

- Requirement: Daily accruals must sum to balance

- Action: Flag discrepancies for review
- Scope: TransferPointsCalculation entity

Validation: 10-YearHistoryCompleteness  
- Requirement: No gaps in 10-year service history
- Source: Cross-reference SuccessFactors + HR Resource Group
- Action: Identify missing periods for data collection
- Scope: ServiceHistory entity

Validation: TransferRatingConsistency
- Requirement: Location ratings must be 1-7 scale
- Frequency: Real-time on data entry
- Action: Reject invalid ratings
- Scope: FOCompany, ServiceHistory entities

- Requirement: Service breaks not exceed 60 school days
- Logic: Calculate break days between assignments
- Action: Set continuousServiceFlag accordingly
- Scope: ServiceHistory entity
```

## Workflow and Business Rule Metadata

### Transfer Application Workflow

```
Workflow: TransferApplicationProcess
- Trigger: Application submission
- Steps:
  1. Initial Validation (Automatic)
  2. Eligibility Check (Automatic)
  3. Regional Review (Manual - HR Consultant)
  4. School Acceptance (Manual - Principal)
  5. Final Approval (Manual - HR Authority)
- Conditions: Service years >= 2, Points balance sufficient
- Notifications: Email to relevant stakeholders at each step
```

### Transfer Point Calculation Business Rules

```
BusinessRule: TransferPointAccrual
- Logic: Daily accrual = (locationRating / 365) * workDays
- Conditions: Full-time equivalent calculation for part-time
- Acceleration: Years >= threshold for ratings 3-7
- Reset: Points reset to 0 for first preference transfers

BusinessRule: ServiceContinuity
- Logic: Track continuous service across locations
- Break Rule: 60 school days max for continuous service
- Temporary Rule: Temporary engagements count if breaks < 60 days
```
